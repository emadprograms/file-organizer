# Phase 24 Plan 04: Fix Propose/Finalize — Full Pipeline in Propose, Programmatic Finalize

## Problem Statement

Waves 2 and 3 implemented `propose` and `finalize` with a fundamentally broken UX:

- **`propose` (Wave 2)** only resolves Area and House. Tenant and Group are left as `U` (Unknown) in the proposed filename. The user sees `Safra D 504 U U 20 مايو 2021 AppendTest_Proposed.pdf` and has no idea which tenant folder or group the file will end up in.
- **`finalize` (Wave 3)** runs the grouping, routing, and generation passes after the user approves. This means the user approved something without knowing the final destination. The entire point of the `_Proposed` system is to tell the user: "this file is going to X place, is that OK?"

This plan fixes both methods so the user always sees exactly where every document will go before approving.

## Step 1: Refactor `propose()` in `src/fs_ui/orchestrator.py`

**File:** `src/fs_ui/orchestrator.py`
**What changes:**

The current `propose` stops after partial inference and renames the ORIGINAL file with incomplete fields. The new `propose` must:

1. **Keep existing logic** for: parsing filename, creating tmp_dir, running `process_unclassified_pdf`, calling `infer_missing_data`, resolving area/house/YAML (lines 77–147 stay mostly the same).

2. **Run the full pipeline** after resolving the house:
   ```python
   # Load YAML
   yaml_data = load_yaml(yaml_path)
   
   # Clean pass
   cleaned_pages, _ = pipeline._clean_documents(tmp_report_path, house_dir, house_id)
   
   # Group pass (always run — no skip)
   raw_pages = [(p.original_index, p) for p in cleaned_pages]
   documents = pipeline._group_documents(raw_pages)
   
   # Route pass (always run — no skip)
   routed_docs = pipeline._route_documents(documents)
   ```

3. **For each routed document**, extract its pages into a separate PDF using `fitz`:
   ```python
   for i, doc in enumerate(routed_docs):
       new_pdf = fitz.open()
       with fitz.open(str(filepath)) as src:
           new_pdf.insert_pdf(src, from_page=doc.start_page, to_page=doc.end_page)
       
       # Resolve ALL 6 fields from the routed document
       tenant = doc.primary_tenant or "UnknownTenant"
       group_folder = doc.folder_path or "UnknownGroup"
       date = pick_date(doc)
       title = doc.brief_arabic_title or "UnknownTitle"
       
       resolved_name = f"{area_id} {house_id} {tenant} {group_folder} {date} {title}"
       proposed_path = inbox / f"{resolved_name}_Proposed.pdf"
       new_pdf.save(str(proposed_path))
       new_pdf.close()
   ```

4. **Save metadata** alongside each proposed file. For each proposed PDF, write a small JSON sidecar in `.tmp_{resolved_name}/` containing:
   - `page_indices`: list of original page indices from the source PDF
   - `cleaned_data`: the cleaned page data for those pages
   - `routed_doc`: the full routed document object
   - `source_house_id`: the house_id
   - `source_area_id`: the area_id
   
   This metadata is used by `finalize` so it knows which pages to append and which JSONs to update.

5. **Delete the original input PDF** after all proposed files are created.

6. **Remove the `resolve_group_mode` / `skip_grouping` / `skip_routing` branching.** The full pipeline always runs. The user's group field in the filename is a *hint*, not a directive to skip passes.

## Step 2: Refactor `finalize()` in `src/fs_ui/orchestrator.py`

**File:** `src/fs_ui/orchestrator.py`
**What changes:**

The current `finalize` runs `run_grouping_pass`, `run_routing_pass`, and `run_generation_pass`. All of that moves into `propose`. The new `finalize` is purely programmatic:

1. **Parse the approved filename** to extract the user's final values for all 6 fields:
   ```
   {area} {house} {tenant} {group} {date} {title} OK.pdf
   ```
   The user may have overridden any field from what was originally proposed.

2. **Resolve destination paths** from the parsed fields:
   - `area_dir = areas_root / area`
   - `house_dir = find_house_dir(area_dir, house)`
   - `tenant_dir = find_or_create_tenant_dir(house_dir, tenant)`
   - `group_folder = resolve_group_folder(group)`  (e.g., "7" → "07_letters")
   - `dest_dir = tenant_dir / group_folder`

3. **Copy and compress the PDF** into the destination folder. Handle filename collisions per D-13 (append timestamp/counter).

4. **Append pages to `_raw_append.pdf`** in `.source_files/`:
   - Read page indices from the sidecar metadata JSON
   - Open `_raw_append.pdf`, get `page_shift = doc.page_count`
   - Append the approved PDF's pages
   - Save

5. **Merge JSONs** into `.source_files/`:
   - Read cleaned data from sidecar, shift `original_index` by `page_shift`, append to `{house_id}_1_cleaned.json`
   - Read routed doc from sidecar, shift `start_page`/`end_page` by `page_shift`, append to `{house_id}_2_grouped.json`
   - Append report entries to `{house_id}_report.json`
   - Write/update `{house_id}_3_routed_and_finalized.json`

6. **Rebuild the finalized PDF** with updated TOC:
   - Read all routed documents from `.source_files`
   - Build TOC entries
   - Save `{house_id}_finalized.pdf` with TOC and compress

7. **Cleanup**: delete the approved PDF from inbox, delete its `.tmp_` sidecar directory.

8. **Zero LLM calls.** Everything here is file operations and JSON manipulation.

## Key Design Decisions

| Decision | Rationale |
|----------|-----------|
| Full pipeline runs in `propose` | The user must see exactly where every document will go before approving. Showing `U U` defeats the purpose of the proposal system. |
| Each `_Proposed` file is independent | No batch tracking needed. User approves, edits, or ignores each file individually. |
| User can override any field before OK | `finalize` reads the filename at approval time, not what was proposed. All JSONs are written based on the approved values. |
| Sidecar metadata in `.tmp_` dirs | `finalize` needs page indices and cleaned data without re-running LLM. A small JSON sidecar per proposed file stores this. |
| No `skip_grouping`/`skip_routing` | The full pipeline always runs. The group field in the filename is a hint for inference, not a directive to skip passes. |
