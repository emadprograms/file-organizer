# Phase 23 - Plan 03: Gap Closure for Tenant Resolution

**Phase:** 23
**Mode:** gap_closure

## Context
During UAT, a major issue was identified: when the `tenant_hint` is provided as `'U'` (Unknown), `resolve_tenant` hardcodes the return value as `'U'`. It fails to utilize the LLM or categorization report to infer the missing tenant name, meaning the system leaves the tenant unresolved instead of attempting to figure it out, which defeats the purpose of the missing data inference layer.

## Objective
Update `infer_missing_data` and `resolve_tenant` to properly extract and resolve the tenant name from the document when the hint is `'U'`, aligning it with how House ID and Date are inferred.

## Instructions
1. **Modify `infer_missing_data` in `src/inbox/resolver.py`:**
   - Update the logic to also check `parsed_cmd.tenant_hint == 'U'`.
   - When processing the `_report.json`, collect the `tenant_name` (or `canonical_tenant`) from the JSON report using a `collections.Counter`, just like `house_counter` and `date_counter`.
   - Add the inferred tenant to the returned dictionary.

2. **Modify `resolve_tenant` in `src/inbox/resolver.py`:**
   - Change the signature to accept the inferred tenant name if available, or update the upstream caller in `src/main.py` to pass the inferred tenant name when `tenant_hint == 'U'`.
   - If `tenant_hint == 'U'` and an inferred tenant name is available, use the inferred name to match against the `tenant_configs` via `canonicalize_with_llm`.
   - Only return `'U'` if inference completely fails or no valid tenant is found.

3. **Update `src/main.py` append mode logic:**
   - Pass the inferred tenant to `resolve_tenant` when processing the `inferred` dictionary from `infer_missing_data`.

4. **Testing:**
   - Add a unit test in `tests/test_resolver.py` to verify that `tenant_hint='U'` successfully extracts and canonicalizes the tenant when a report exists.
