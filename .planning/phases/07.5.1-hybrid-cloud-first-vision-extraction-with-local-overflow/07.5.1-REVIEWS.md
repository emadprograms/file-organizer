# Cross-AI Plan Review — Phase 07.5.1

## Gemini Review

# Plan Review: Phase 07.5.1 - Hybrid Cloud-First Vision Extraction

## Summary
The plan is a targeted refactor of `src/llm.py` designed to restore the preferred cloud model (`gemma-4-26b-a4b-it`) and implement a non-blocking overflow mechanism to a local model (`qwen2.5vl:7b`) during rate-limit cooldowns. It correctly identifies the need to scale RPM limits based on the number of available API keys and emphasizes the removal of blocking `time.sleep()` calls to ensure the GUI and pipeline remain responsive during the cloud-to-local transition.

## Strengths
- **Precise Targeting:** The plan focuses exclusively on `src/llm.py`, minimizing the risk of regression in other modules.
- **Model Alignment:** Explicitly restores the user's requested model (`gemma-4-26b-a4b-it`), correcting previous drifts to `gemini-1.5-flash`.
- **Resource Optimization:** The dynamic scaling of the RPM limit (`15 * len(self.api_keys)`) is a logical improvement that maximizes the utility of provided credentials.
- **Latency Focus:** The insistence on `no_wait=True` and the removal of `time.sleep()` directly addresses the core "bottleneck" pain point described in the context.

## Concerns
- **Passive Logic Enforcement (MEDIUM):** Task 3 is framed as an "audit" and "confirmation" rather than an "implementation." If the audit reveals that the non-blocking fallback logic is missing or broken, the plan does not explicitly define the necessary code changes to fix it. This creates a risk where the agent might "confirm" a failure but not apply the fix.
- **Cooldown State Management (LOW):** While the research mentioned a `cloud_cooldown_until` timestamp, the plan does not explicitly task the agent with implementing or verifying the logic that *clears* the cooldown after 65 seconds. If the state isn't managed precisely, the system might stay in "local mode" longer than necessary or attempt cloud calls too early, triggering another 429.
- **RPM Limit Assumption (LOW):** The plan assumes RPM limits are tied strictly to keys. If the API provider imposes an IP-based limit, scaling the limit by the number of keys will not prevent 429s and may lead to more aggressive rate-limiting.

## Suggestions
- **Convert Audit to Action:** Reformulate Task 3. Instead of "confirming" behavior, the task should be: *"Ensure `extract_page` implements a strict try-except block that catches 429 errors and immediately invokes `_extract_text_with_qwen` without any intervening sleep calls."*
- **Explicit Cooldown Logic:** Add a requirement to verify the timestamp-based cooldown mechanism. Ensure there is a clear check: `if current_time < cooldown_expiry: use_local()`.
- **Logging Visibility:** Add a small task to ensure that when the system switches from Cloud $\rightarrow$ Local and Local $\rightarrow$ Cloud, it logs this transition clearly. This is vital for debugging the "seamless snap back" requirement.

## Risk Assessment
**Overall Risk: LOW**

The changes are surgical and contained within a single file. The primary risk is functional (failure to actually achieve non-blocking behavior) rather than systemic (crashing the app). By converting the "audit" task into a "fix" task, the risk is effectively mitigated.

## Ollama Review

## Review of Phase 07.5.1: Hybrid Cloud-First Vision Extraction with Local Overflow

### Summary
The plan sets out to implement a hybrid vision extraction model that uses `gemma-4-26b-a4b-it` as the primary cloud-based OCR service and switches to the local Qwen model during rate limit exhaustion. The tasks are straightforward but critical for ensuring non-blocking fallback logic and dynamic scaling of request limits based on available API keys.

### Strengths
- **Clear Goal Alignment**: Each task directly addresses a specific requirement (D-01 through D-05) laid out in the project context.
- **Focused Refactoring**: The refactoring tasks are confined to existing code, ensuring no new classes or functions are introduced unnecessarily.
- **Dynamic Scaling**: Allows for dynamic scaling of RPM based on available API keys without hardcoded limits.

### Concerns
- **Error Handling:** There's an assumption that error handling in `extract_page` is already optimal. It would be beneficial to explicitly verify this through unit tests rather than relying solely on code walkthrough (Severity: MEDIUM).
- **State Management**: The existing state management mechanism might not cater well to handling multiple requests during cooldown periods, which could lead to partial failures or inconsistent application states (Severity: HIGH).
- **Unit Testing**: Detailed unit testing is essential but not explicitly mentioned in the tasks. Without adequate test coverage, the refactoring may introduce unforeseen bugs (Severity: MEDIUM).

### Suggestions
- **Explicit Error Handling:** Add explicit error handling and fallback logic verification in `extract_page` through additional unit tests when catching 429 errors.
- **State Manager Enhancements:** Introduce or enhance a robust state manager that tracks cooldown periods and capacity, ensuring seamless handoff between cloud and local models during high load conditions.
- **End-to-End Testing:** Include end-to-end testing scenarios with synthetic API responses simulating both regular flow and overflow conditions to ensure the system behavior is as expected.

### Risk Assessment
**Overall Risk Level: MEDIUM**

Justification:
The primary concerns revolve around error handling and consistency during cooldown periods, which are fundamental to the hybrid model's operation. While the refactoring appears straightforward, overlooking edge cases like precise state management could lead to systemic issues. Comprehensive testing is paramount; thus, additional steps to ensure robust fallback logic through unit tests would mitigate these risks significantly.

## Consensus Summary

Both reviewers agree that the plan is well-targeted and effectively addresses the core requirements for hybrid extraction (scaling RPM by API keys and non-blocking fallback). However, both note that the plan leaves some critical operational details underspecified, particularly around how the system manages the "cooldown" state and handles errors during the transition.

### Agreed Strengths
- **Surgical Refactoring:** The focus is correctly constrained to `src/llm.py`.
- **Dynamic RPM Scaling:** The strategy to calculate RPM as `15 * len(api_keys)` is solid.
- **Goal Alignment:** Directly addresses the project's latency and throughput bottlenecks by removing blocking sleep calls.

### Agreed Concerns
- **Passive/Missing State Management (HIGH):** The plan lacks concrete instructions for managing the 65-second cooldown state and ensuring the system elegantly hands off requests to the local model during this period without dropping data.
- **Vague Error Handling (MEDIUM):** Task 3 is currently framed as an "audit" rather than a strict implementation requirement. Relying on an audit rather than explicit error handling and try-except logic creates a risk of failure.

### Divergent Views
- **Testing Approach:** Ollama strongly advocates for introducing unit and end-to-end testing with synthetic API responses to validate the fallback logic. Gemini considers the risk low enough that code changes and clear logging are sufficient.
