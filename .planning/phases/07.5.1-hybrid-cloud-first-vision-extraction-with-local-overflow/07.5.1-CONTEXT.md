# Phase 07.5.1: Hybrid Cloud-First Vision Extraction with Local Overflow - Context

**Gathered:** 2026-06-25
**Status:** Ready for planning

<domain>
## Phase Boundary

Redesign the Pass 1a (Vision Extraction) hybrid cloud/local approach. The system must use `gemma-4-26b-a4b-it` (the user's preferred cloud model) for fast and accurate extraction. When the cloud API hits its rate limit (15 requests/min), the system enters a 65-second cooldown. Instead of blocking the thread, the system must utilize this 65-second waiting period to process pages using the local model (`qwen2.5vl:7b`). Once the cloud cooldown expires, execution should seamlessly snap back to `gemma-4-26b-a4b-it`.
</domain>

<decisions>
## Implementation Decisions

### Vision Model Architecture (Pass 1a only)
- **D-01:** **Primary Cloud Model:** Use `gemma-4-26b-a4b-it` for the primary Pass 1a OCR.
- **D-02:** **Cooldown Overflow:** When the cloud model hits the 15-request rate limit, a 65-second cooldown begins. During this specific window, instantly fall back to the local `qwen2.5vl:7b` model to continue processing pages locally.
- **D-03:** **Resumption:** The moment the 65-second cooldown clears, seamlessly revert to processing the remaining pages with `gemma-4-26b-a4b-it`.
- **D-04:** **Fix Existing Bottlenecks:** The previous implementation used blocking `time.sleep()` which defeated the purpose of the fallback. The new logic must ensure non-blocking handoffs to the local model during the cooldown window.
- **D-05:** **Dynamic Key Handling:** Properly utilize all API keys provided in `.env` without artificially throttling the global rate limit to 15 if multiple valid keys exist (unless explicitly desired to share a single IP limit).

</decisions>
