# Phase 07.6: Semantic Rolling Window Grouping - Research

## Goal
Replace the fragmented metadata-based grouping logic in Pass 2 with a semantic, "story-based" approach using a rolling window comparison of page summaries.

## What do I need to know to PLAN this phase well?

### 1. Existing Data Structures & Pass 1 Updates
- Pass 1 uses `src/llm.py` to extract `PageClassification` which contains a `summary` field. We need to ensure the `_build_system_prompt()` explicitly instructs the LLM to write **verbose, descriptive summaries** in the `summary` field (e.g. "This is a form for [Person] regarding [Topic], specifically explaining X", or "This page is blank and illegible").
- Remove `is_continuation` from the Pass 1 LLM prompt and the system entirely, since the semantic window will replace this legacy hint.
- `PageClassification` in `src/schemas.py` already has a `summary` field. Its description should be updated to align with the new D-03/D-04 decisions (verbose description of what exactly is going on, allowing fallback phrasing if the page is blank).

### 2. The Semantic Rolling Window Logic
- Instead of grouping strictly by `category`, `names`, `dates`, and `is_continuation`, Pass 2 (`_group_pages_into_documents` in `src/pipeline.py`) will evaluate adjacencies using a **rolling window**.
- When processing `page N`, we want to decide if it belongs to the `current_document` (which ends at `page N-1`).
- To make this decision, we will invoke a Local LLM (Qwen2.5 14b) passing the accumulated context of the current document (e.g., the summaries of the last `k` pages) and the summary of `page N`.
- The LLM will return a boolean `is_continuation` and a `reason` for the decision.
- **Exceptions & Interruptions (D-02, D-05):**
  - If a page is entirely blank or just a signature ("BS" page), its summary will say so. The grouping LLM prompt must be instructed to *carry over* the group across such interruptions without breaking the document.

### 3. API & Prompt Changes
- **Pass 1 Prompts (`src/llm.py`):**
  - Update `_build_system_prompt()` to heavily emphasize creating a detailed, verbose `summary`. Remove mentions of `is_continuation` from the required JSON schema.
- **New Pass 2 LLM Call (`src/llm.py`):**
  - Create a new method, e.g., `check_semantic_continuation(current_doc_summaries: list[str], next_page_summary: str) -> SemanticMatchResult`.
  - Use the Local LLM via OpenAI API (Qwen2.5 14b). The cost and latency are acceptable for this per-page-pair comparison.
  - Define a new schema in `schemas.py` for this response, e.g., `SemanticMatchResult(is_continuation: bool, reason: str)`.

### 4. Architectural Flow
1. **Pass 1:** Extract `PageClassification` including the new verbose `summary`.
2. **Pass 1.5:** Entity resolution (no change to the existing code).
3. **Pass 2:** `_group_pages_into_documents(raw_pages, canonical_mapping_clean)`
   - Initialize `documents` list.
   - For each page, evaluate its primary tenant using the canonical mapping as before.
   - For grouping: Call `llm.check_semantic_continuation(previous_summaries, current_summary)`.
   - If `True` (or if the new page is just a blank/junk page that carries over), extend `end_page` of the current document group.
   - If `False`, create a new `DocumentGroup` and reset the rolling window.

## Validation Architecture

To validate the semantic rolling window logic efficiently without running the massive pipeline every time, we will use a targeted testing strategy.

### 1. Isolated Unit Tests for Semantic Grouping
- Create a test script (e.g., `tests/test_semantic_grouping.py`) that mocks Pass 1 outputs.
- Provide arrays of mocked `PageClassification` objects with pre-written `summary` strings representing known edge cases:
  - **Case A (Normal Continuation):** Page 1 summary: "This is a contract preamble for Ali." Page 2 summary: "This is the signature page for the contract." -> **Should group.**
  - **Case B (Interruption):** Page 1: "This is page 1 of an allocation letter for Sara." Page 2: "This page is blank and illegible." Page 3: "This is page 2 of the allocation letter for Sara." -> **Should group all 3 together.**
  - **Case C (Junk Page):** Page 1: "This is an EWA notification for Khalid." Page 2: "This page contains only an illegible signature." -> **Should group together.**
  - **Case D (Hard Boundary):** Page 1: "This is the final page of a maintenance request." Page 2: "This is a new personal details form for Ahmed." -> **Should split into two documents.**

### 2. Integration Testing on Real PDFs
- Run the pipeline on specific test PDFs (e.g., a small multi-page sample) that have known blank pages or signature-only pages in the middle.
- Observe the emitted `.cache.json` to verify that Pass 1 correctly generated verbose summaries and properly identified blank/junk pages.
- Verify the final console output/logs to ensure that the document groups correctly span across the blank pages and split exactly where the topic changes.

### 3. Traceability
- The new `check_semantic_continuation` LLM call must return a `reason` alongside the boolean.
- The pipeline must log this `reason` (e.g., `[Semantic Window] Grouping page 4 with doc 1 because: {reason}`) so developers can instantly audit *why* the LLM made a specific grouping decision.
