# Plan Review: Phase 07.6 - Semantic Rolling Window Grouping

## Summary
The plan is well-structured and directly addresses the core objective: moving from fragmented metadata-based grouping to a "story-based" semantic approach. It correctly identifies the necessary changes across the schema, LLM interface, and pipeline logic, and includes a strong, case-driven validation strategy. The alignment with the specific user decisions (D-01 through D-05) is excellent.

## Strengths
- **Strong Validation Strategy:** The use of mocked `PageClassification` objects to test specific edge cases (blank pages, junk signatures, hard boundaries) ensures the logic is verified without requiring expensive full-pipeline runs.
- **Clean Legacy Removal:** Explicitly removing the `is_continuation` field and the Vision OCR footer code prevents "ghost logic" from persisting in the codebase.
- **Traceability:** Including the `reason` field in `SemanticMatchResult` and logging it in the pipeline is critical for debugging the non-deterministic nature of LLM grouping.
- **Adherence to Constraints:** The plan strictly follows the mandate to use a local LLM for comparison and encourages verbose, natural-language summaries.

## Concerns
- **Undefined Window Size (MEDIUM):** The plan mentions "summaries of the last `k` pages" but does not define `k`. If `k` is too small, the LLM loses context of the "story"; if it's too large (e.g., the entire current document), the prompt will bloat and latency will increase as the document grows.
- **Tenant Conflict Handling (MEDIUM):** Pass 1.5 (Entity Resolution) happens before this grouping. The plan doesn't explicitly state how to handle a case where the semantic window says "continue," but the canonical mapping shows a sudden change in the primary tenant. A change in person usually constitutes a hard boundary regardless of semantic flow.
- **Per-Page Latency (LOW):** While the LLM is local, adding one LLM call per page transition in a 200-page document introduces significant sequential latency to Pass 2.
- **Error Fallback (LOW):** There is no defined behavior for when `check_semantic_continuation` fails (e.g., API timeout or malformed JSON).

## Suggestions
- **Define `k`:** Establish a fixed window size (e.g., `k=3` or `k=5`) for the rolling summaries to maintain consistent prompt size and performance.
- **Integrate Tenant Checks:** In `_group_pages_into_documents`, the logic should check if the primary tenant has changed. If the tenant changes, it should likely trigger a split *unless* the semantic LLM explicitly identifies it as a joint document.
- **Implement a Fallback:** Add a simple try-except block around the semantic check. If it fails, the safest default is likely to treat the page as a new document to prevent merging unrelated content.
- **Prompt Hint for "Joint Documents":** In the `check_semantic_continuation` prompt, explicitly tell the LLM how to handle pages that transition between related family members to avoid unnecessary splits.

## Risk Assessment
**Risk Level: LOW**

**Justification:** The technical changes are surgical and the validation plan is comprehensive. The primary risks are performance-related (latency) and edge-case related (tenant switches), both of which are manageable through prompt tuning and simple conditional logic in `src/pipeline.py`.
