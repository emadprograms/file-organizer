# Phase 19.1: Fix YAML Integration Architecture - Context

**Gathered:** 2026-07-16
**Status:** Ready for planning

<domain>
## Phase Boundary

Refactoring the pipeline to properly utilize `yaml_loader.py` for configuration loading and remove dead code, properly connecting it to `pipeline.py` instead of bypassing it in `phase.py`.

</domain>

<decisions>
## Implementation Decisions

### YAML file naming convention
- **D-01:** Use `{house_id}_tenants.yaml` to support multiple houses in the same source directory. This maintains the current implementation's behavior.

### YAML schema structure
- **D-02:** Update `yaml_loader.py` to parse the complex schema (list of dicts with name, start_date, end_date) that `phase.py` is currently expecting.

### Missing YAML fallback orchestration
- **D-03:** `yaml_loader.py` should return `None` or an empty list when the file is missing, signaling `pipeline.py` to gracefully fallback to anchor logic.

</decisions>

## Existing Code Insights

### Reusable Assets
- `src/tenant_config/yaml_loader.py`: The entry point for configuration loading, which must be updated.
- `src/pipeline/pipeline.py`: The orchestrator that must now invoke `yaml_loader.py` and coordinate fallback.
- `src/timeline/phase.py`: The file currently bypassing `yaml_loader.py` directly, whose logic must be migrated/cleaned.

### Established Patterns
- LLM client injection into the pipeline is well-established. Configuration and fallback routing will just provide `allowed_tenants` or timelines for further processing.

### Integration Points
- Connecting `yaml_loader.py`'s response into `pipeline.py` and subsequently into the `_group_documents` or `_route_documents` logic (via `phase.py`'s existing cleaning logic, which currently runs via `process_cleaning_phase`).
