# Phase 24.1 Plan Review

## Summary
The proposed plans effectively adopt the new, authentic fixtures to create deterministic, isolated tests. Transitioning from fuzzy stdout checks to hardcoded path assertions (Decision D-01) and mapping each inbox scenario to a dedicated test function (Decision D-04) are strong architectural improvements. However, there are critical misunderstandings regarding how the `FSUIOrchestrator` and `create` mode CLI handle state and paths, which will cause the tests to fail if implemented exactly as planned.

## Strengths
- **Targeted Testing:** Replacing sequential, monolithic tests with dedicated test functions for each inbox PDF (Plan 24.1-02-T01) greatly improves test isolation and maintainability.
- **Deterministic Assertions:** Asserting against exact disk structures rather than fuzzy string matching is an excellent strategy.
- **Offline Reliability:** Pre-generating `_report.json` mock files (Plan 24.1-01) ensures the E2E suite remains fast and deterministic without hitting the live Gemini API.

## Concerns
- **HIGH:** In Plan 24.1-02-T01, instructing the test to "copy the corresponding `_report.json`... to the inbox alongside the target PDF" will not bypass the LLM. When `orchestrator.propose()` is called, it creates a hidden `.tmp_{stem}_master/` directory and calls `process_unclassified_pdf()`. The categorization logic looks for the report *inside* the temporary directory or `.source_files/`, not in the raw inbox folder (`src/categorization/categorization.py:56-64`). Consequently, the system will attempt to call the LLM using the dummy API key, causing the test to crash.
- **HIGH:** Plan 24.1-02-T02 proposes running `main.py create` on the new `1273` and `504` golden structures. However, these structures represent *already finalized* houses containing massive `_finalized.pdf` archives (`tests/fixtures/e2e/golden_state/areas/Safra C/1273 - يونس محمد ملاك/1273_finalized.pdf`). Because `create` mode is designed for raw inputs and `categorization.py` only ignores `_categorized` files (not `_finalized` ones), it will attempt to extract and process the entire finalized archive as if it were a raw input, leading to severe test slowdowns or crashes.

## Suggestions
- **Modify Plan 24.1-02-T01:** Instead of copying the mock report next to the PDF in the inbox, have the test pre-create the hidden directory (e.g., `<inbox>/.tmp_SAFC 1273 U G U_master/`) and copy the `_report.json` inside it before invoking `orchestrator.propose()`. This perfectly aligns with where `process_unclassified_pdf` expects the file to be.
- **Modify Plan 24.1-02-T02:** Do not test `create` mode using the finalized golden state directories. Either create a distinct `raw_input` fixture specifically for testing `main.py create`, or test the `append` mode logic instead if you must use the finalized areas.
- **System Improvement:** Update `src/categorization/categorization.py` (around line 48) to explicitly skip processing any PDF whose name contains `_finalized`, just as it currently skips `_categorized` PDFs. This prevents accidental re-processing of archives in edge cases.

## Risk Assessment
**HIGH.** The plans are conceptually sound, but the specific implementation details regarding file placement and CLI modes will cause immediate test failures. Adjusting the mock injection strategy and the `create` mode targets as suggested will fully mitigate these risks.

## Peer Review Cycle 2

# Phase 24.1 Implementation Plan Review

**Summary**
The plan effectively targets the right mechanisms for testing the pipeline without relying on the live LLM API. Injecting mock reports into the `.tmp_{stem}_master/` directory is an elegant way to bypass `categorization.py`'s processing, and skipping `_finalized.pdf` files is a sound optimization. However, there are critical mechanical issues regarding how `append` mode instantiates the LLM client and an inherited data structure flaw from Plan 01 that will cause the tests to crash.

**Strengths**
- Placing the injected mock report inside `.tmp_{stem}_master/` correctly satisfies both the bypass condition in `categorization.py:61` (`(target_dir / "_report.json").exists()`) and the explicit requirement in `orchestrator.py:93` (`master_tmp_dir / f"{filepath.stem}_report.json"`).
- Adding `or "_finalized" in pdf_path.name` in `categorization.py` is a direct and sound fix to prevent reprocessing large finalized archives.

**Concerns**
- **HIGH** — **`append` mode ignores `--skip-llm` flag:** T02 proposes running `main.py append` to test the pipeline. However, `append` mode instantiates its own `LLMClient` (`src/cli/main.py:99`) and does NOT expose or inherit the `--skip-llm` argument. As a result, operations in `FSUIOrchestrator` like `resolve_tenant` will invoke `canonicalize_with_llm`, which will attempt a real network call with the dummy API key and fail. 
- **HIGH** — **Inherited dictionary structure bug from Plan 01:** Plan 01 specifies creating the mock `_report.json` files as dictionaries keyed by page strings. However, `orchestrator.py:254` slices the parsed JSON: `doc_report = full_report_data[start_idx:end_idx+1]`. Attempting to slice a dictionary will raise a `TypeError: unhashable type: 'slice'`. The mocks MUST be JSON lists.
- **MEDIUM** — **Loss of `create` mode CLI coverage:** By entirely replacing `tests/test_pipeline_e2e.py` to focus on `append` mode, end-to-end coverage for the `create` command and its flags (`--dry-run`, etc. in `main.py`) will be lost.

**Suggestions**
- **Patch `skip_llm` in T01 and T02:** Since `append` mode and `e2e_workspace` do not naturally skip LLM calls, use `unittest.mock.patch` to explicitly mock the `skip_llm` property to `True` for `LLMClient` in these tests.
- **Handle `SystemExit` correctly:** When mocking `time.sleep` to raise `sys.exit(0)` to break the infinite listener loop, ensure the call to `main()` is wrapped in `with pytest.raises(SystemExit):` to prevent the exception from crashing the test runner.
- **Ensure correct mock filename format:** When copying the mock report into `.tmp_{stem}_master/`, verify it is copied under the exact name `{stem}_report.json`. While `categorization.py` flexibly accepts `_report.json`, `orchestrator.py` strictly looks for the prefixed version.
- **Fix Plan 01 Mocks:** Correct the mock `_report.json` assets to be arrays (lists) of objects rather than dictionaries.
- **Retain a `create` mode test:** Retain at least one E2E test for the `create` mode using the new 1273 golden structures to ensure that pathway doesn't regress.
