# Phase 07.4: Harden Prompt for Document Detection (INSERTED) - Execution Plan 01

## Objective
Implement schema updates, prompt hardening with Arabic subject patterns, and conditional LLM routing (falling back to Gemma 4 26b) for document classification when local LLM lacks confidence.

## Tasks

### Task 01: Schema Updates
**Description:** Update the data models to support a fallback signal from the local LLM.
**Target File:** `src/schemas.py`
**Actions:**
1. In the `PageClassification` model, add a new boolean field `needs_gemma_fallback: bool = Field(default=False, description="Set to true if there is no Subject and no strong pattern match")`.

### Task 02: Prompt Hardening for Local & Fallback LLMs
**Description:** Modify the system prompts to enforce strict Arabic subject patterns, restrict 'amar takhsees' false positives, and add local fallback rules.
**Target File:** `src/llm.py`
**Actions:**
1. Update the system prompts (both local and Gemma fallback) to ALWAYS analyze the subject first.
2. Add the specific pattern rules for Amar Takhsees, Maintenance (ensure ANY mention of "inspection" goes to maintenance), House Modifications, Key Handover, EWA, Allowance, Rent Deduction, Notifications, and Basic vs Personal Details.
3. **CRITICAL (Amar Takhsees):** Update both the local and Gemma fallback prompts with a very strict definition of `amar_takhsees`. Explicitly state: "An allocation order (amar takhsees) is STRICTLY an order from a higher authority or someone important claiming or telling to give the person (primary tenant) a place to stay. Do NOT classify random documents as allocation orders."
4. **CRITICAL (Rent vs Allowance):** Explicitly instruct both models that Rent Deduction letters will ALWAYS contain "30 bd" or "60 bd". Use this presence/absence to strictly disambiguate Rent Deduction from Allowance Deduction (وقف استقطاع بدل الانتفاع), which will NOT have these BD amounts.
5. **CRITICAL (Basic vs Personal Details):** Explicitly instruct both models on the difference: `basic details` are strictly forms about the person. `personal details` refers to pictures of identity cards, passports, and other non-form documents related to the person and his family. Anything related to the person and his family that is NOT a form goes into `personal details`.
6. Instruct the local LLM: If NO subject and none of the strong patterns match, set `needs_gemma_fallback = true` and do NOT guess blindly.

### Task 03: Inference Routing and Fallback Logic
**Description:** Update the local inference function to respect the `needs_gemma_fallback` flag.
**Target File:** `src/llm.py`
**Actions:**
1. In `classify_page()` (or the relevant method handling local inference), inspect the parsed result from the local LLM.
2. If `parsed_result.needs_gemma_fallback` is True, log a message (e.g., `[Local Inference Refused] No subject/strong pattern found. Falling back to gemini-4-26b.`).
3. Route the request to the Gemma fallback execution path instead of returning the local LLM's incomplete result.

### Task 04: Validation Infrastructure - Unit & Integration Tests
**Description:** Create tests to verify the schema, prompt fallback flag, and routing logic.
**Target File:** `tests/test_vision.py` (or new test file `tests/test_llm_fallback.py`)
**Actions:**
1. Mock test local LLM responses to verify `needs_gemma_fallback` triggers Gemma fallback correctly in `classify_page()`.
2. Add assertions for the pattern logic if dummy PDFs or text inputs are available for testing.
