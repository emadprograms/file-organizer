## 1. What are we trying to achieve?
The goal is to improve the accuracy of the local LLM (Qwen2.5-VL) by forcing it to rely on specific "Subject" (الموضوع) patterns and specific keywords, rather than blindly guessing from the body text. If the local LLM cannot find a subject or a strong matching pattern, it should gracefully refuse to guess and intentionally trigger a fallback to the more capable Gemma 4 26b model.

## 2. Where does the relevant logic live?
- **Prompt definition:** `src/llm.py` -> `GemmaClient._build_system_prompt()`. This method currently generates a single shared prompt used by both the local LLM and the Gemma fallback.
- **Inference logic:** `src/llm.py` -> `GemmaClient.classify_page()`. Currently, it attempts to run the local Qwen model. If an exception occurs (e.g., connection error, Pydantic validation failure), it catches it and falls back to Gemma.
- **Data Schemas:** `src/schemas.py` -> `PageClassification` and `Category`.

## 3. What needs to change to implement this well?

To implement Phase 07.4 correctly, you will need to plan for three major modifications:

### A. Schema Updates (`src/schemas.py`)
We need a structured way for the local LLM to signal "I don't have enough context, please fall back."
- **Recommended Approach:** Add a new boolean field to the `PageClassification` schema:
  `needs_gemma_fallback: bool = Field(default=False, description="Set to true if there is no Subject and no strong pattern match")`
- **Alternative:** Add `UNKNOWN_NEEDS_GEMINI` to the `Category` enum. (This is slightly riskier as it pollutes the 13-category constraint).

### B. Prompt Hardening (`src/llm.py`)
The system prompt needs to be heavily modified to include the new logic. 
*Note: Because Qwen and Gemma currently share the same prompt, you may want to separate them (e.g., `_build_local_system_prompt()` vs `_build_gemma_system_prompt()`), or instruct the model: "If you are unsure, set needs_gemma_fallback to true. But if you are the fallback model, do your absolute best to categorize."*

The local prompt MUST include the specific decision rules from `07.4-CONTEXT.md`:
1. **Rule 1 (Subject First):** Always look for the Subject (الموضوع) or Sender/Receiver first.
2. **Rule 2 (Personal Details):** If the document is an ID card, civil record, etc., classify as `personal_details` immediately.
3. **Rule 3 (Specific Patterns):** 
   - Exact subject "الموضوع : الوحدات السكنية" or mentions "تمديد الإقامة / السكن" -> `amar_takhsees`
   - Sender/receiver is "إدارة الأشغال" or yellow paper with inspection -> `maintenance`
   - Subject contains "طلب" and mentions modifying house -> `modifications`
   - Contains "استمارة تسليم الوحدات السكنية التابعة لوزارة الداخلية" -> `key_handover_form`
   - Contains meter number (e.g., "الوحدة السكنية رقم ( 508 ) طريق 4411") -> `ewa_related_letters`
   - Subject is "الموضوع: وقف استقطاع بدل الانتفاع" -> `allowance_deduction`
   - Contains rent deduction text (e.g., "استقطاع الإيجار") -> `rent_deduction`
   - Contains "إشعار" or "اشعار" -> `notifications`
4. **Rule 4 (Fallback Triggers):** If NONE of the above match, and there is no explicit Subject, set `needs_gemma_fallback = true`. Do not guess blindly.

## 4. Risks and Considerations for the Planner
- **Prompt Size & Complexity:** The local model (Qwen 7b) might get confused by too many negative constraints. Keep the prompt instructions highly structured, perhaps using a decision-tree format in the prompt text.
- **Testing:** The Phase requires "stress testing". Ensure the Plan includes steps to run unit tests (`pytest`) or a scratch script against specific dummy PDFs (ones with subjects and ones without) to verify that the `needs_gemma_fallback` flag is accurately toggled and respected.

## Validation Architecture

To ensure the prompt hardening and fallback logic work correctly, the validation strategy should encompass:

1. **Unit Testing Local Inference Triggers:**
   - Create mock responses or use existing dummy PDFs to assert that documents *without* a subject and *without* strong patterns successfully set `needs_gemma_fallback = True`.
   - Assert that documents *with* recognized patterns (like "إدارة الأشغال") return `needs_gemma_fallback = False` and the correct `Category`.

2. **Integration Testing Fallback Routing:**
   - In `test_vision.py` or a new integration test script, mock the local LLM to return `needs_gemma_fallback = True`.
   - Assert that `classify_page()` intercepts this flag and correctly triggers the `gemini-4-26b` fallback, ultimately returning the parsed Gemma response instead of the refused local response.

3. **Stress Testing with Real Documents:**
   - Run a batch of known complex, highly-nuanced Arabic documents that previously confused the local LLM.
   - Verify that these documents are correctly routed to the Gemma model instead of being randomly miscategorized by the local LLM.
   - Verify that standard letters (like 'amar_takhsees' and 'ewa_related_letters') are correctly caught by the hardened prompt and handled locally without needing Gemma.
