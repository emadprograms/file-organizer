# Phase 16.1 Context & Decisions

## Overview
This phase addresses the confusing checkpoint and intermediate state management across the different passes of the pipeline. The previous architecture scattered JSON files across the root output directory and a `checkpoints/` folder with inconsistent naming and cleanup logic (e.g., `cleaned.json`, `grouped.json`, `manifest.json`).

## Decisions Made

1. **Naming Convention:**
   All intermediate files will be renamed to explicitly reflect the pipeline pass they belong to:
   - `{house_id}_1_cleaned.json` (Replaces old `cleaned.json`)
   - `{house_id}_2_grouped.json` (Replaces old `grouped.json` and `{house_id}_runs_checkpoint.json`)
   - `{house_id}_3_routed_and_finalized.json` (Replaces the final `manifest.json`)
   
2. **Consolidated Storage Location:**
   All intermediate state files will be stored in a single unified `.run_cache/` folder (replacing the old `checkpoints/` folder).

3. **Final Output and Cleanup:**
   Instead of deleting the intermediate JSON files, when the pipeline completes successfully, we will move `{house_id}_1_cleaned.json`, `{house_id}_2_grouped.json`, and `{house_id}_3_routed_and_finalized.json` from `.run_cache/` into the final `source_files/` directory. 
   
   **Rationale**: This preserves the state at each pass permanently. If a user encounters an error in routing later or wants to redo a specific stage, they can simply delete the specific JSON file (e.g., delete `{house_id}_3_routed_and_finalized.json`) and the script will automatically resume from the previous valid checkpoint (e.g., `{house_id}_2_grouped.json`) without having to start over from cleaning.

## Open Areas for the Planner / Researcher
