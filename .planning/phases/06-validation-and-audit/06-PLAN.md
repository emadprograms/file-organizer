# Phase 06: Validation and Audit - Execution Plan

## Goal
Verify that the logging overhaul is 100% compliant with the new standards, ensuring no legacy logging patterns remain and that the telemetry system behaves predictably across all verbosity levels.

## 1. Audit Phase (Structural)
This phase focuses on static analysis of the codebase to ensure consistency and compliance.

### 1.1 Print Statement Audit
- **Task:** Search for any remaining `print()` calls in `src/` that are not `console.print`.
- **Method:** Use `grep` with a regex that excludes `console.print`.
- **Success Criteria:** Zero matches for standard `print(` in `src/`.

### 1.2 Logger Initialization Audit
- **Task:** Verify that every module in `src/` and `tests/` initializes the logger using the canonical pattern: `logger = logging.getLogger(f"file_organizer.{__name__}")`.
- **Method:** Grep search for logger assignments and verify the naming convention.
- **Success Criteria:** All modules use the same naming pattern and the variable name `logger`.

### 1.3 Exception Handling Audit
- **Task:** Ensure all `except` blocks that log errors use `logger.exception()` instead of `logger.error()`.
- **Method:** Search for `except` blocks and check for the accompanying log call.
- **Success Criteria:** Stack traces are preserved via `logger.exception` in all critical error paths.

## 2. Validation Phase (Functional)
This phase involves running the application to verify runtime behavior.

### 2.1 Verbosity Mode Validation
- **Task:** Run the application in both `verbose=False` (default) and `verbose=True` modes.
- **Validation:**
    - **`verbose=False`**: Verify that third-party noise (`openai`, `google-genai`, `urllib3`, `httpcore`) is suppressed (WARNING+ only).
    - **`verbose=True`**: Verify that `file_organizer` hierarchy logs are prominent and the strict whitelist is applied.
- **Success Criteria:** Log volume and content match the expected verbosity levels.

### 2.2 Log File Integrity
- **Task:** Validate the structure and content of the output logs.
- **Validation:**
    - **`app.log`**: Confirm it contains human-readable INFO+ logs.
    - **`debug.log`**: Confirm it is a valid JSONL file containing DEBUG+ logs.
- **Success Criteria:** Files are correctly formatted and contain the appropriate log levels.

### 2.3 Telemetry and Traces Validation
- **Task:** Verify that the decision-trace system is functioning.
- **Validation:**
    - Check `traces.jsonl` for correctly formatted decision-trace payloads generated by `log_decision_trace`.
- **Success Criteria:** `traces.jsonl` is populated with a logical sequence of system decisions.

### 2.4 Error Path Testing
- **Task:** Intentionally trigger known error paths (e.g., invalid API keys, missing files).
- **Validation:**
    - Verify that `debug.log` contains the full stack trace for these errors.
- **Success Criteria:** `debug.log` provides sufficient detail for root-cause analysis.

## 3. Finalization

### 3.1 Audit Report
- **Task:** Generate a final `audit_report.txt` summarizing:
    - Results of the print audit.
    - Confirmation of logger initialization consistency.
    - Confirmation of JSONL validity for `debug.log`.
    - Verification of noise suppression behavior.
- **Location:** `.planning/phases/06-validation-and-audit/audit_report.txt`

- **Task:** Mark Phase 06 as complete.
- **Actions:**
