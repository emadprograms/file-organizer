# Phase 04.1 Research: Refactor core modules

## Objective
Answer: "What do I need to know to PLAN this phase well?"

## 1. Scope & Boundaries
- **Strictly Refactoring**: This phase introduces **no new capabilities**. It is focused entirely on reducing bloat, increasing testability, and modularity before the testing phase begins.
- **Context**: Local LLM logic and API key switching complexity have already been removed (Phase 01 and 02). The system is now exclusively cloud-based (Gemini -> OpenRouter -> Groq).

## 2. Key Architectural Decisions to Implement
The planner needs to structure tasks around three major refactoring efforts:

### A. Utility Extraction (`organizer.py` -> `utils.py`)
- Extract date parsing, string normalization, and filename sanitization logic from `organizer.py` into a new `src/utils.py`.
- **Goal**: Keep `organizer.py` focused strictly on file operations.

### B. Strategy Pattern for LLMs (`llm.py`)
- Replace the monolithic `_route_llm_call` and its internal retry/fallback routing with distinct **Provider classes** (e.g., `GeminiProvider`, `OpenRouterProvider`, `GroqProvider`) using the Strategy Pattern.
- **Goal**: Isolate provider-specific logic and simplify the unified routing interface exposed by `llm.py`.

### C. Class-based Extractors (`pipeline.py`)
- Decompose the massive `process_pdf` method into distinct class-based extractors (e.g., `VisionExtractor`, `CloudExtractor`).
- **Goal**: Separate orchestration logic. `pipeline.py` will serve as the orchestrator wiring together these Extractor classes.
- **Constraint**: `SimpleCache` in `pipeline.py` must be preserved and slightly adapted to work with the new Extractor classes.

## 3. Interfaces & Dependencies
- **Data Models**: Use existing Pydantic models from `src/schemas.py` (e.g., `PageClassification`, `DocumentGroup`) to guide the input/output interfaces of the new Extractor and Provider classes.
- **Integration Points**:
  - `organizer.py` will now depend on `src/utils.py`.
  - `llm.py` remains the primary entry point for LLM calls, but delegates to the new Provider classes.
  - `pipeline.py` remains the orchestrator, leveraging the new Extractor classes.

## 4. Recommended Execution Order for the Plan
To minimize breakage during refactoring, the plan should ideally follow a bottom-up approach:
1. **Step 1**: Create `src/utils.py` and refactor `organizer.py`. (Independent leaf node)
2. **Step 2**: Refactor `src/llm.py` to implement the Provider strategy classes. (Used by pipeline)
3. **Step 3**: Refactor `src/pipeline.py` to implement the Extractor classes and wire up the new LLM providers. (Top-level orchestrator)

## 5. Constraints & Exclusions
- **No Test Generation**: Tests are explicitly deferred to a later phase (TEST-01). The plan should not include writing automated unit tests.
- **Cloud-Only Focus**: Do not attempt to add or consider local model logic. Focus purely on the `google-genai` and `openai` clients.
