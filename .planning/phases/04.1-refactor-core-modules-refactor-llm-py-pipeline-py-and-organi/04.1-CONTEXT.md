# Phase 04.1: Refactor core modules - Context

**Gathered:** 2026-06-28
**Status:** Ready for planning

<domain>
## Phase Boundary

Refactoring the core logic (`llm.py`, `pipeline.py`, `organizer.py`) for testability, modularity, and reducing bloat before adding tests. No new capabilities are added in this phase.

</domain>

<decisions>
## Implementation Decisions

### Pipeline Orchestration
- **D-01:** Separate the orchestration logic in `pipeline.py` into distinct class-based extractors (e.g., VisionExtractor, CloudExtractor) instead of functional decomposition within a single massive `process_pdf` method.

### LLM Fallback Logic
- **D-02:** Use the Strategy Pattern with separate Provider classes (e.g., `GeminiProvider`, `OpenRouterProvider`, `GroqProvider`) in `llm.py` to handle provider-specific logic, extracting the retry and fallback routing out of the monolithic `_route_llm_call`.

### Organizer Utilities
- **D-03:** Extract the date parsing, string normalization, and filename sanitization logic from `organizer.py` into a new dedicated `utils.py` module to keep the Organizer focused strictly on file operations.

</decisions>

## Existing Code Insights

### Reusable Assets
- `src/schemas.py`: Pydantic models (e.g., `PageClassification`, `DocumentGroup`) that will guide the interfaces of the new Extractor and Provider classes.

### Established Patterns
- **Cloud-Only APIs**: Local LLM logic has already been removed; focus entirely on the `google-genai` and `openai` clients.
- **Atomic Caching**: `SimpleCache` in `pipeline.py` should be preserved or slightly adapted when moving to Extractor classes.

### Integration Points
- `pipeline.py` will serve as the orchestrator that wires together the new Extractor classes.
- `llm.py` will expose a unified LLM routing interface backed by the new Provider classes.
- `organizer.py` will import from the new `utils.py`.
