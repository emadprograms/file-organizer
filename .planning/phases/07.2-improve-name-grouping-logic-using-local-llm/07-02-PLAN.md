# Phase 07.2: Improve name grouping logic using local LLM

## Goal
Refactor name grouping logic in `src/pipeline.py` to utilize the local LLM (`Qwen2-VL-7B-Instruct`) for semantic matching, keeping exact string matching as a fast path. If Qwen fails or times out, fallback to the existing Gemma model logic.

## Context
During entity resolution, we group document pages to existing resident timelines based on the primary tenant's name. Currently, this relies heavily on strict word intersection. Moving forward, ambiguous cases will be sent to the local LLM for a semantic decision.

## Requirements Mapped
- `SEMANTIC-01`: Fast-Path Exact String Matching
- `SEMANTIC-02`: Qwen2-VL Name Matching Fallback
- `SEMANTIC-03`: Gemma 4 26b Secondary Fallback
- `SEMANTIC-04`: Structured JSON Outputs

## Step-by-Step Implementation Plan

### Step 1: Define LLM Name Matcher Method
- Add a new method in `src/llm.py` (e.g., `check_name_match`) that prompts the local `Qwen2-VL` model.
- Decorate this method with `@lru_cache(maxsize=1024)` from `functools` to cache results and prevent redundant LLM invocations for identical name pairs across the batch.
- The prompt must include the two names and the document category.
- **Prompt Engineering:** Explicitly instruct the LLM to ignore common Arabic prefixes (like "ال") and titles (like "السيد") when determining a semantic match to avoid false negatives.
- The prompt must enforce a structured JSON response: `{"is_match": true/false, "reason": "..."}`.
- Include a broad `try/except` block to fallback to `GemmaClient` if Qwen fails. This must explicitly catch network errors (`ConnectionError`, `TimeoutError`, `requests.exceptions.RequestException`) in case the local inference server is down, alongside JSON decoding errors.

### Step 2: Refactor Grouping Logic in `pipeline.py`
- Locate the entity resolution block in `src/pipeline.py` (around line 200).
- Preserve the existing `len(words_current.intersection(words_candidate)) >= min(...)` check as the first pass (Fast Path).
- If the exact match returns `False`, call the new `check_name_match` LLM method.
- If the LLM returns `{"is_match": true}`, group the documents together.
- Update both anchor and non-anchor processing branches to use this dual-check mechanism.

### Step 3: Testing & Validation
- Run the pipeline on a small subset of documents known to have name variations (e.g., "خالد" vs "خالد محمد").
- Verify that the Fast Path triggers for exact matches (zero latency).
- Verify that the local LLM triggers for variations and groups them correctly.
- Force an LLM failure to ensure the Gemma fallback triggers successfully without halting the pipeline.
