# Cross-AI Plan Review — Phase 07.2

## Gemini Review

# Plan Review: Phase 07.2 - Improve Name Grouping Logic using Local LLM

## 1. Summary
The plan is a sound, tiered approach to solving the "entity resolution" problem for Arabic names. By implementing a "Fast Path" (exact match) $\rightarrow$ "Local LLM" (semantic match) $\rightarrow$ "Cloud/Heavy LLM" (fallback) pipeline, the design balances performance with accuracy. The focus on structured JSON outputs and specific file targets makes the implementation path clear and surgical.

## 2. Strengths
*   **Performance-First Design:** The use of a fast-path intersection check ensures that the system doesn't introduce unnecessary latency for clear matches.
*   **Resilience:** The multi-layered fallback mechanism (Qwen $\rightarrow$ Gemma) ensures that the pipeline remains functional even if the local inference server fails or returns malformed data.
*   **Auditability:** Requiring a `reason` field in the JSON output is excellent for debugging why two names were grouped together, which is critical for a high-stakes document organization task.
*   **Surgical Scope:** The changes are limited to `src/llm.py` and `src/pipeline.py`, minimizing the risk of regressions in other parts of the system.

## 3. Concerns
*   **Inference Bottleneck (MEDIUM):** If a document has many resident candidates and few exact matches, the system will make numerous sequential calls to the local LLM. This could significantly increase processing time per file.
*   **Arabic Name Nuances (MEDIUM):** Arabic names often vary by the presence of "ال" (Al-), common titles (e.g., "السيد"), or slight spelling variations in common names. The plan doesn't explicitly specify if the prompt will guide the LLM on these specific linguistic rules.
*   **Redundant Computations (LOW):** The plan doesn't mention caching. If the same name pair appears across multiple documents or pages, the LLM will be asked to resolve the same pair repeatedly.
*   **Over-Grouping Risk (LOW):** Semantic matching can sometimes be "too helpful," grouping people with similar but distinct names. There is no mention of a confidence threshold or a way to flag "uncertain" matches for human review.

## 4. Suggestions
*   **Implement a Result Cache:** Introduce a simple in-memory cache (or a dictionary) in `src/llm.py` to store the results of `(name_a, name_b)`. This prevents redundant LLM calls for common name pairs.
*   **Batching Requests:** Instead of one call per name pair, modify `check_name_match` to accept a *list* of candidate pairs and return a list of JSON results. This reduces the overhead of multiple LLM invocations.
*   **Refine Prompt Guidelines:** Explicitly instruct the LLM in the prompt to ignore common Arabic prefixes/suffixes and to be conservative with matches to avoid "over-grouping."
*   **Logging for Audit:** Ensure that the `reason` provided by the LLM is logged to the `gui_debug.log` or a similar file, allowing the user to verify the semantic logic after processing.

## 5. Risk Assessment
**Overall Risk: LOW**

**Justification:** 
The plan is an enhancement of an existing functional system. Because the "Fast Path" is preserved, existing functionality is guaranteed to remain intact. The primary risks are performance degradation (latency) and a slight increase in false-positive groupings, both of which can be mitigated via batching, caching, and prompt tuning.

## Antigravity Review

# Plan Review: Phase 07.2

## 1. Summary
The plan demonstrates a robust approach to modernizing the pipeline by bridging deterministic logic with probabilistic LLM-based entity resolution. Maintaining backward compatibility via the Fast Path is an excellent architectural choice.

## 2. Strengths
*   **Non-destructive Enhancement:** Maintaining the `len(words_current.intersection)` check guarantees that performance remains lightning-fast for standard cases.
*   **Graceful Degradation:** The fallback from a local `Qwen2-VL` model to the established `GemmaClient` acts as a perfect safety net.

## 3. Concerns
*   **Exception Scope (LOW):** The plan mentions using a `try/except` block to catch Qwen JSON failures, but network timeouts or `ConnectionRefusedError` (if the local server dies) must also be explicitly caught so the pipeline doesn't crash entirely.

## 4. Suggestions
*   **Trivial Memoization:** I highly agree with Gemini's cache suggestion. In Python, you can simply apply `@lru_cache(maxsize=1024)` to the `check_name_match` function (or a dictionary cache on the class) to gain massive speedups for recurring family members across hundreds of pages.

## 5. Risk Assessment
**Overall Risk: LOW**

**Justification:**
This is a low-risk, high-reward feature. If the local LLM misbehaves, the existing codebase logic and Gemma fallbacks offer strong guarantees of continuity.

## Consensus Summary

Both Gemini and Antigravity strongly approve of the Phase 07.2 plan, noting that the "Fast Path" exact-matching design ensures zero regressions for currently working edge cases. The multi-tiered fallback architecture (Exact -> Local LLM -> Cloud LLM) is praised for its resilience.

### Agreed Strengths
- The Fast-Path integration preserves zero-latency performance for clear exact matches.
- The fallback structure guarantees pipeline resilience against local server downtime.
- Forcing a JSON output with a `reason` field provides essential debugging capabilities for entity resolution decisions.

### Agreed Concerns
- **Redundant LLM Invocations:** The plan currently does not mention caching LLM results, which could slow down the pipeline if the same ambiguous name pair is encountered multiple times.
- **Arabic Nuances in Prompts:** The LLM prompt must explicitly be guided to handle Arabic prefixes ("ال") to avoid making the same mistakes as the old deterministic logic.

### Divergent Views
None. Both models agree on the low-risk nature of the plan and heavily emphasize adding an in-memory cache to the LLM matching function to solve potential bottleneck issues.
